포트폴리오

홍길동을 소개합니다

홍길동은 복잡한 요구사항을 사용자가 자연스럽게 이해할 수 있는 화면으로 바꾸는 프론트엔드 개발자입니다. 기능이 동작하는 데서 멈추지 않고, 사용자가 서비스를 처음 마주하는 순간부터 작업을 완료하는 순간까지 흐름이 끊기지 않는 경험을 만드는 일을 중요하게 생각합니다.

제품의 목적과 운영 환경을 먼저 이해한 뒤 화면 구조, 데이터 흐름, 상태 변화와 예외 상황을 함께 설계합니다. 디자이너의 의도를 세밀하게 구현하면서도 장기적인 유지보수성을 놓치지 않으며, 서버 개발자와는 API 계약과 오류 처리 원칙을 일찍 맞춰 개발 과정에서 발생할 수 있는 불확실성을 줄입니다.

기술 스택

웹 표준을 기반으로 한 견고한 구현을 우선합니다. 특정 도구에 화면 구조를 종속시키기보다 HTML의 의미, CSS의 레이아웃 원리, JavaScript의 실행 모델을 이해하고 프로젝트의 규모와 팀 상황에 맞는 기술을 선택합니다.

  • CoreHTML5, CSS3, JavaScript, TypeScript를 사용해 시맨틱 구조와 예측 가능한 인터랙션을 구현합니다.
  • FrontendReact와 Next.js를 중심으로 컴포넌트 구조, 라우팅, 서버 렌더링과 클라이언트 상태를 설계합니다.
  • StateTanStack Query, Zustand, Context API를 데이터의 성격에 따라 구분해 서버 상태와 UI 상태를 관리합니다.
  • BackendNode.js와 REST API에 대한 이해를 바탕으로 인증, 캐시, 오류 응답과 비동기 데이터 흐름을 다룹니다.
  • QualityVitest, React Testing Library, Playwright로 핵심 사용자 흐름과 회귀 위험이 높은 기능을 검증합니다.
  • DeliveryGit, GitHub Actions, Vercel, Docker를 활용해 리뷰 가능한 변경 단위와 반복 가능한 배포 흐름을 만듭니다.

프로젝트

홍길동의 프로젝트는 화려한 기능의 수보다 해결한 문제와 개선된 사용자 경험을 중심으로 설명합니다. 각 작업에서 무엇을 관찰했고, 어떤 선택을 했으며, 그 결과를 어떻게 확인했는지 명확하게 남깁니다.

팀 업무 관리 서비스

프로젝트별 업무와 일정이 여러 채널에 흩어져 진행 상황을 파악하기 어렵다는 문제를 해결했습니다. Next.js와 TypeScript로 화면을 구성하고, 서버 상태는 TanStack Query로 관리해 목록 갱신과 낙관적 업데이트의 기준을 일관되게 만들었습니다.

업무 생성과 상태 변경처럼 사용 빈도가 높은 흐름은 키보드만으로도 완료할 수 있도록 설계했습니다. 데이터가 많아지는 상황에서는 페이지 전체를 다시 그리지 않도록 렌더링 범위를 줄였고, 사용자가 입력한 내용이 네트워크 오류로 사라지지 않도록 복구 가능한 임시 상태를 적용했습니다.

콘텐츠 탐색 플랫폼

검색 결과가 늦게 나타나고 필터를 변경할 때마다 화면이 크게 흔들리는 경험을 개선했습니다. URL을 검색 조건의 기준으로 삼아 새로고침과 링크 공유 이후에도 같은 결과를 재현할 수 있게 했으며, 중복 요청을 줄이고 이전 결과를 유지해 화면 전환의 안정감을 높였습니다.

이미지와 비핵심 UI는 필요한 시점에 불러오고, 주요 콘텐츠의 크기를 미리 확보해 레이아웃 이동을 줄였습니다. 접근성 검사와 실제 키보드 탐색을 함께 수행해 시각적 완성도뿐 아니라 다양한 사용 환경에서도 동일한 기능을 제공했습니다.

문제 해결 방식

문제를 발견하면 바로 코드를 바꾸기보다 먼저 재현 조건과 영향을 받는 범위를 정리합니다. 네트워크 요청, 렌더링 시간, 브라우저 이벤트와 사용자 행동을 분리해 관찰하고, 개선 전후를 같은 조건에서 비교할 수 있는 기준을 만듭니다.

성능 문제는 Lighthouse 점수 하나로 판단하지 않습니다. 실제 화면에서 중요한 콘텐츠가 언제 보이는지, 입력에 얼마나 빠르게 반응하는지, 불필요한 JavaScript가 사용자 기기에서 어떤 부담을 만드는지를 함께 확인합니다. 해결책 역시 라이브러리 추가보다 데이터 구조 단순화, 렌더링 범위 축소, 캐시 정책 정리처럼 원인에 가까운 변경을 우선합니다.

예측 가능한 실패를 설계합니다

좋은 화면은 성공한 상황만 자연스러운 화면이 아닙니다. 로딩, 빈 결과, 권한 부족, 네트워크 단절과 서버 오류를 별도의 예외가 아니라 제품 흐름의 일부로 취급합니다. 사용자가 현재 상태와 다음 행동을 이해할 수 있도록 메시지와 복구 수단을 함께 제공합니다.

협업과 품질

팀이 빠르게 움직이려면 개인의 구현 속도보다 변경을 이해하고 검증하기 쉬운 구조가 중요하다고 생각합니다. 작업을 작은 단위로 나누고, Pull Request에는 변경 이유와 확인 방법, 남아 있는 위험을 함께 기록합니다. 리뷰에서는 취향보다 사용자 영향, 일관성, 복잡도와 운영 비용을 기준으로 의견을 나눕니다.

  • 접근성의미 있는 마크업, 명확한 포커스, 키보드 조작과 충분한 대비를 기본 구현 조건으로 둡니다.
  • 테스트구현 세부사항보다 사용자가 실제로 수행하는 핵심 흐름과 실패 복구 과정을 검증합니다.
  • 문서화결정의 배경과 제약을 남겨 다음 작업자가 같은 문제를 처음부터 다시 풀지 않도록 합니다.
  • 관찰배포 이후 오류와 성능 지표를 확인하고, 실제 사용 패턴을 다음 개선의 근거로 연결합니다.

앞으로

홍길동은 프론트엔드를 화면 구현에 한정하지 않고 제품과 시스템 사이를 연결하는 역할로 바라봅니다. 브라우저 렌더링, 네트워크, 서버 사이드 렌더링과 웹 성능에 대한 이해를 계속 넓히며, 사용자가 체감하는 품질과 팀이 유지할 수 있는 구조를 함께 개선하고자 합니다.

새로운 기술을 빠르게 도입하는 것보다 문제에 적합한 기술을 선택하고 그 선택을 설명할 수 있는 개발자를 지향합니다. 작고 명확한 개선을 반복하며 오래 사용할 수 있는 제품을 만드는 것이 홍길동의 다음 목표입니다.
